# ViMax Lite と本家 ViMax の仕様差分

このドキュメントは、現在の ViMax Lite 実装と、本家 [HKUDS/ViMax](https://github.com/HKUDS/ViMax) の仕様差分を整理したものです。

調査日: 2026-05-21

## 位置づけ

ViMax Lite は、本家 ViMax の完全コピーではありません。動画生成 API や有料画像生成を前提にせず、動画生成直前までの制作設計、RAG、エージェント設計、画像生成プロンプト、参照画像管理を学習・ポートフォリオ化するための実装です。

本家 ViMax は、アイデアや脚本から最終動画までを目指すエンドツーエンドのエージェント型動画生成フレームワークです。README では、脚本生成、絵コンテ、キャラクター作成、参照画像管理、整合性検証、動画生成までを自動化する構成として説明されています。

ViMax Lite は、次の範囲を現在の到達点にしています。

- アイデア入力または脚本入力から制作設計を生成する。
- Pydantic モデルで構造化された `design.json` を生成する。
- キャラクター、シーン、ショット、画像生成プロンプト、動画生成プロンプトを出力する。
- 簡易 RAG ストアにキャラクター、ショット、プロンプト、画像メタデータを保存する。
- Web UI からアイデア入力、生成進捗確認、参照画像アップロード、ショット画像アップロードを行う。
- ChatGPT などの手作業画像生成に使える、参照画像つきの手順書を出力する。
- ChatGPT に貼り付ける画像生成プロンプトは英語で出力し、UI や学習説明は日本語で表示する。

## 全体アーキテクチャの差分

| 項目 | 本家 ViMax | 現在の ViMax Lite |
|---|---|---|
| 主目的 | アイデア、脚本、小説、参照画像から動画を生成する | 動画生成直前までの制作設計と画像生成準備を行う |
| 実行形態 | Python スクリプト中心。設定ファイルでモデル、画像生成、動画生成を指定 | Python CLI と FastAPI Web UI |
| 入力 | idea、script、novel、reference images、style、config | idea、script、audience、style、duration_seconds |
| 出力 | フレーム、画像、動画クリップ、最終動画、ログ、作業ディレクトリ成果物 | Markdown、JSON、RAG trace、画像プロンプト、動画プロンプト、参照画像計画、手作業画像生成手順書 |
| 動画生成 | 実装対象。first frame / last frame から動画生成する | 実装しない。動画生成プロンプトまで |
| 画像生成 | 画像生成 API を使い、参照画像つきで候補画像を生成する | Gemini 画像生成または手作業 ChatGPT 生成を想定。手作業フローを重視 |
| 整合性維持 | 参照画像選択、複数候補生成、VLM によるベスト画像選択 | キャラクター参照画像、直前ショット画像、RAG メモリ、人間確認で補助 |
| 学習用途 | 研究・実用寄り | RAG、エージェント、構造化出力、生成AI活用の学習とポートフォリオ化を重視 |

## 参照画像ワークフローの差分

### 本家 ViMax

本家 ViMax は、参照画像の扱いを自動化しています。README では、長い動画になるほどキャラクター、背景、画角、時系列の整合性が崩れやすい問題に対して、参照画像選択と整合性チェックを行うと説明されています。

コード上では、主に次の部品が関係します。

- `CharacterPortraitsGenerator`
  - キャラクターの正面画像を生成する。
  - 正面画像を参照して側面画像を生成する。
  - 正面画像を参照して背面画像を生成する。
- `ReferenceImageSelector`
  - 現在生成したいフレームの説明を受け取る。
  - キャラクター参照画像や過去フレーム画像の候補から、必要な参照画像を選ぶ。
  - 候補が多い場合はテキスト説明で事前に絞り込む。
  - その後、マルチモーダルモデルで実画像も見て最終選択する。
  - 最大 8 枚程度の参照画像を選ぶ。
  - 直近の過去フレームを優先する。
  - 同じキャラクターの正面、側面、背面が重複する場合は、現在ショットに最も合う向きだけを選ぶ。
  - 参照画像のどの要素を使うべきかを含む画像生成プロンプトも生成する。
- `BestImageSelector`
  - 参照画像、目標説明、複数の生成候補画像を受け取る。
  - Character Consistency、Spatial Consistency、Description Accuracy を比較する。
  - 最も整合性が高い候補画像を選ぶ。
- `CameraImageGenerator`
  - カメラ位置の親子関係を推定する。
  - 広い画角から狭い画角へ、または関連するカメラ間で参照をつなぐ。
  - transition video から次のカメラ用画像を取り出す処理も持つ。

つまり本家は、単に「基本参照画像 + 直前画像」を添付するだけではありません。現在ショットに必要な参照画像を、キャラクター、環境、スタイル、構図、時系列の観点から選び直します。

### 現在の ViMax Lite

現在の ViMax Lite は、人間参加型の参照画像ワークフローです。ChatGPT の Web UI を自動操作するのではなく、Web UI 上で「何を ChatGPT に添付するべきか」を管理します。加えて、任意でローカルSDXLとIP-Adapterを使い、採用済み参照画像を視覚条件として渡した候補生成もできます。

現在の流れは次の通りです。

1. Web UI または CLI で制作設計を生成する。
2. キャラクター参照シートを生成する。
3. ChatGPT で `front / side / back / detail` の参照画像を手作業生成してアップロードするか、SDXLで候補を生成して確認後に採用する。
4. SDXLを使う場合、採用済み `front` をIP-Adapterへ渡して他の向きの整合性を補助する。
5. ショット生成キューで、各ショットに添付する参照画像とプロンプトを確認する。
6. ChatGPT に参照画像を添付して表示されたプロンプトを貼り付けるか、SDXLへ参照画像を条件付けして候補を生成する。
7. アップロードした画像、または確認して採用したSDXL候補を正式なショット画像として保存する。
8. 次ショットでは、キャラクター参照画像に加えて直前ショット画像も参照候補に入る。

現状の `build_shot_reference_plan()` は、次のルールで参照画像を組みます。

- シーンに登場するキャラクターごとに、`front` と `side` の参照画像を入れる。
- 2枚目以降のショットでは、直前ショットの生成画像 `images/manual/<previous_shot_id>.png` を入れる。
- 参照画像の選択はスコアリングではなく、決め打ちルールで行う。
- SDXL生成時は、存在する参照画像をIP-Adapterへ視覚条件として渡す。
- SDXLの生成結果は `images/sdxl_candidates/` に置き、人間が採用するまで正式な参照やショットにはしない。

そのため、現在の ViMax Lite は次の形です。

```text
ショット1:
  キャラクター基本参照画像 + ショット1プロンプト

ショット2:
  キャラクター基本参照画像 + ショット1生成画像 + ショット2プロンプト

ショット3:
  キャラクター基本参照画像 + ショット2生成画像 + ショット3プロンプト
```

## 現在できていること

### 制作設計

- `vimax-lite idea2design` でアイデアから制作設計を生成できる。
- `vimax-lite script2design` で脚本から制作設計を生成できる。
- `mock` provider で API キーなしの検証ができる。
- `gemini` provider で実 API を使った生成ができる。

### 構造化出力

- Pydantic モデルを使って、制作設計を `design.json` に保存する。
- Markdown 版の `design.md`、`storyboard.md`、`image_prompts.md`、`video_prompts.md` も出力する。

### RAG

- キャラクター、ショット、画像プロンプト、生成画像メタデータを `rag_store.json` に保存する。
- 後続エージェントで使った検索結果を `rag_trace.md` に出力する。
- 現在の検索は簡易的なキーワード一致であり、ベクトル検索や埋め込み検索ではない。
- 現在のトークン化は英数字中心なので、日本語検索品質には改善余地がある。

### Web UI

- Web UI からアイデアを入力できる。
- バックグラウンドで制作設計を生成し、進捗バーとイベントログを表示できる。
- プロジェクト画面でショット数、画像生成済み数、残り生成枚数を確認できる。
- 参照画像ページで、キャラクター参照画像用のプロンプトを確認し、生成画像をアップロードできる。
- ショット生成ページで、各ショットの添付参照画像、ChatGPT 貼り付け用プロンプト、アップロードフォームを確認できる。
- 追加依存関係と実行デバイスが利用可能な場合、SDXL候補を生成し、画面で確認して採用できる。
- 参照画像のあるSDXL生成では、IP-Adapterを使って画像データ自体を条件付けに渡す。

### 手作業画像生成

- `references/character_reference_sheet.md` を生成する。
- `reference_plan.md` と `reference_plan.json` を生成する。
- `manual_generation_guide.md` を生成する。
- 生成画像を `images/manual/<shot_id>.png` に保存する。
- アップロードされた画像を `images/image_manifest.json` に記録する。

### ローカルSDXL候補生成

- SDXL候補画像と生成条件を `images/sdxl_candidates/` に保存する。
- 候補の採用操作を行ったときだけ、正式画像とmanifestを更新する。
- 画像比率に応じた生成サイズをショット設計から引き継ぐ。
- 依存関係未導入時や生成失敗時には、ジョブを失敗状態として表示する。

## まだ本家に届いていないこと

### 参照画像選択

本家は、現在ショットに必要な参照画像をモデルで選びます。ViMax Lite は、現時点では固定ルールです。

未実装:

- 参照画像候補をスコアリングする。
- キャラクターの向きに応じて `front / side / back` のうち最適な参照だけを選ぶ。
- 同じ情報を持つ参照画像を重複排除する。
- 直近ショットを優先しつつ、必要なら過去の重要ショットも選ぶ。
- 最大 8 枚程度に参照画像を絞る。
- 参照画像のどの要素を使うかをプロンプト内に明示する。

### 画像候補の自動評価

本家は、複数の候補画像を生成し、VLM/MLLM で最も整合性が高い画像を選びます。ViMax Lite は、ChatGPTからのアップロードまたはSDXL候補をユーザーが見て採用します。

未実装:

- 1ショットにつき複数候補画像を比較管理する（現状は最新のSDXL候補1枚を保持する）。
- 候補画像を Character Consistency、Spatial Consistency、Description Accuracy で評価する。
- 評価理由を保存する。
- 最良候補を自動選択する。

### カメラ構造

本家は、複数カメラの親子関係を推定し、広い画角や近い画角の関係を使って整合性を保ちます。ViMax Lite は、ショット順に直前画像を参照するだけです。

未実装:

- camera tree の構築。
- 親カメラ、子カメラ、依存ショットの推定。
- 広い画角から狭い画角への参照継承。
- transition video から次カメラの初期画像を取り出す処理。

### 動画生成

本家はAI動画生成まで行います。ViMax Lite は動画生成 API を使わず、制作設計に加えてRemotionによる静止画連結動画を生成できます。MVモードではアップロードしたBGMも組み込めます。

未実装:

- first frame / last frame から動画生成する。
- AI動画クリップを生成して結合する。
- SEや読み上げ音声を生成して最終タイムラインへ組み込む。

### 自動リトライとフォールバック

本家は、複数のエージェントや生成処理にリトライを組み込んでいます。ViMax Lite も一部の provider エラー処理はありますが、全体としてはまだ軽量です。

未実装:

- ショット単位の再生成ループ。
- 整合性評価に基づく Revision Agent の自動再実行。
- 画像生成失敗時の代替モデル切り替え。
- Web UI 上での再試行操作。

## 仕様差分を一言で説明するなら

ViMax Lite は、本家 ViMax の「制作設計、構造化出力、RAG、参照画像管理、整合性維持」という考え方を、動画生成 API なしで学べるように縮小実装したものです。

ただし、画像の整合性維持については、単なるプロンプト管理に留めず、次のような本家に近い構造をすでに持っています。

- キャラクター参照画像を先に用意する。
- 各ショットに添付すべき参照画像を明示する。
- 生成後の画像を次ショットの参照に回す。
- RAG に設計情報と画像メタデータを保存する。
- Web UI で残り生成枚数と作業状態を管理する。
- 画像生成モデルへ渡す最終プロンプトは英語にし、参照画像の役割を `Image 0`、`Image 1` のように明示する。
- ローカルSDXL利用時にはIP-Adapterへ参照画像そのものを渡し、生成候補を確認してから正式採用する。

一方で、本家の強みである「モデルによる参照画像選択」「複数候補画像の自動評価」「camera tree」「動画生成」は、今後の拡張領域です。

## 次に本家へ近づける実装ステップ

### Step 1: 参照画像の見える化

- ショット生成ページに参照画像サムネイルを表示する。
- 参照画像が未アップロードなら、未準備として表示する。
- 直前ショット画像がない場合、次ショットを「生成非推奨」と表示する。

### Step 2: Reference Image Selector Lite

- 候補参照画像を `character_reference`、`previous_shot`、`scene_memory` に分類する。
- ショット説明、カメラ、登場人物、背景、時系列から簡易スコアを付ける。
- 最大 8 枚に絞る。
- なぜその参照画像を選んだかを `reference_plan.json` に保存する。

### Step 3: ChatGPT貼り付けプロンプトの強化

- 「Image 0 はキャラクター外見」「Image 1 は直前ショットの背景と構図」のように、参照先の役割を明記する。
- 本家 `ReferenceImageSelector` のように、参照画像ごとの利用箇所をプロンプトに含める。

### Step 4: 候補画像レビュー

- 1ショットに複数候補画像をアップロードできるようにする。
- 手動評価チェックリストを保存する。
- その後、Gemini Vision などで自動評価できる構造にする。

### Step 5: Best Image Selector Lite

- 候補画像を Character Consistency、Spatial Consistency、Description Accuracy で採点する。
- 選択理由を `continuity_report.md` に追記する。
- 最良画像だけを次ショット参照に採用する。

### Step 6: Camera Tree Lite

- ショットの camera 情報から、wide、medium、close-up の親子関係を推定する。
- 直前ショットだけでなく、同じカメラ系列や親カメラの画像も参照候補に入れる。

## ポートフォリオでの説明例

このプロジェクトは「動画生成そのもの」ではなく、「動画生成を安定させる前段の制作設計エージェント」を実装したものです。

本家 ViMax のようなエンドツーエンド動画生成では、キャラクターや背景の一貫性が大きな課題になります。そこで ViMax Lite では、キャラクター参照画像、過去ショット画像、RAG メモリ、構造化 JSON、整合性レポートを使い、手作業の ChatGPT 画像生成でも破綻しにくいワークフローを作りました。

現在は人間参加型ですが、設計上は Reference Image Selector、Best Image Selector、Vision-based Critic に拡張できるように分けています。そのため、RAG、エージェント、構造化出力、マルチモーダル生成、評価ループを段階的に学べる構成になっています。
